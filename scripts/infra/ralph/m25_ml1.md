## 2. FINDINGS — Real Bias Risks

### CRITICAL

**None identified.** The architecture correctly separates validation (for threshold optimization) from test (for honest OOS reporting).

### HIGH

#### H-1: Threshold Sweep Multiple Testing on Validation Set
**File:** meta_label.py, lines 73-106  
**Location:** `_optimize_threshold_profit()` 

The function sweeps ~40 thresholds (default `THRESHOLD_MIN=0.30` to `THRESHOLD_MAX=0.90` step 0.02 = 31 values, plus additional logic in `_optimize_threshold`). Selecting the best of 31+ thresholds on validation data creates multiple testing bias within the validation set.

**Mitigating factors:**
- The 4-gate quality system on the frozen test set provides an additional backstop
- The threshold is optimized on `val_delta_r` (total R), not Sharpe (noisier with small samples)
- Per-session models require `n_session >= effective_min` samples before threshold search runs

**Severity:** HIGH because selecting from 31+ thresholds inflates false positive rate, but MITIGATED by the test-set quality gates that run AFTER threshold selection.

#### M-2: Session Split Uses Global Index Boundaries
**File:** meta_label.py, lines 237-245

**Potential issue:** Sessions are split at GLOBAL data boundaries (60%/80% of total dataset), not at session-specific boundaries. If a session has few samples early in the dataset and many later, the train/val/test ratios within that session may deviate significantly from 60/20/20.

**Mitigation:** Lines 229-235 check `len(val_idx) < 20 or len(test_idx) < 20` and skip training if splits are too small.

**Severity:** MEDIUM - acceptable because MIN_SESSION_SAMPLES gates ensure adequate split sizes, but could cause imbalanced splits for low-frequency sessions.

#### M-3: Single Config Selection May Introduce Implicit Selection Bias
**File:** meta_label.py (in load_single_config_feature_matrix, not shown but referenced at lines 175-181)

When `single_config=True`, the code selects one configuration per (day, session) using either `max_samples` (most data) or `best_sharpe` (highest Sharpe). 

**Concern:** If `max_samples` always picks the configuration with the most trades, this may systematically select for:

**Severity:** MEDIUM - the config selection logic is domain-aware and intentional, but creates implicit selection that isn't corrected by BH FDR (which happens downstream in strategy_validator.py).
