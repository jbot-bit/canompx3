## § Ch 3 — Labeling (verbatim load-bearing passages, PDF p. 70-78)

### § 3.1-3.3 The fixed-time horizon method and why it fails (verbatim, PDF p. 70-71)

> "As it relates to finance, virtually all ML papers label observations using the fixed-time horizon method. […] An observation Xi is assigned a label yi ∈ {−1, 0, 1}, [where the label is +1 if return exceeds +τ, −1 if it falls below −τ, 0 otherwise, with τ a pre-defined constant threshold]."

> "Despite its popularity, there are several reasons to avoid this approach in most cases. First, as we saw in Chapter 2, time bars do not exhibit good statistical properties. Second, the same threshold τ is applied regardless of the observed volatility. […] In other words, it is a very common error to label observations according to a fixed threshold on time bars."

> "But even these two improvements miss a key flaw of the fixed-time horizon method: the path followed by prices. Every investment strategy has stop-loss limits, whether they are self-imposed by the portfolio manager, enforced by the risk department, or triggered by a margin call. It is simply unrealistic to build a strategy that profits from positions that would have been stopped-out by the exchange. That virtually no publication accounts for that when labeling observations tells you something about the current state of the investment literature."

**Project mapping:** the labels in our `orb_outcomes` table are not fixed-time-horizon — they are triple-barrier-equivalent (E2 stop-market = lower horizontal barrier; RR target = upper horizontal barrier; horizon end-of-session = vertical barrier). The project is already structurally compatible with LdP's preferred labeling framework. The relevant flag is: any new research script that constructs custom labels for ML or filter-overlay work MUST adopt the triple-barrier framing, not fixed-time-horizon thresholds.

### § 3.4 The Triple-Barrier Method — definition (verbatim, PDF p. 72)

> "Here I will introduce an alternative labeling method that I have not found in the literature. […] I call it the triple-barrier method because it labels an observation according to the first barrier touched out of three barriers. First, we set two horizontal barriers and one vertical barrier. The two horizontal barriers are defined by profit-taking and stop-loss limits, which are a dynamic function of estimated volatility (whether realized or implied). The third barrier is defined in terms of number of bars elapsed since the position was taken (an expiration limit). If the upper barrier is touched first, we label the observation as a 1. If the lower barrier is touched first, we label the observation as a −1. If the vertical barrier is touched first, we have two choices: the sign of the return, or a 0."

> "You may have noticed that the triple-barrier method is path-dependent. In order to label an observation, we must take into account the entire path spanning [ti,0, ti,0 + h], where h defines the vertical barrier (the expiration limit)."

**Project mapping:** maps directly onto our E2 trade ledger. `pnl_r` = horizon return at first barrier touch. `mfe_r` / `mae_r` track the path. The path-dependence point is critical: any feature derived from the post-entry path (break_bar_volume, break_dir, etc.) is by LdP's framing a **post-treatment variable**, not a predictor. This is the same insight as the project's E2 break-bar look-ahead registry (`feedback_e2_lookahead_drift_check_landed.md`). The triple-barrier framing makes the look-ahead boundary explicit.

### § 3.6 Meta-labeling (verbatim, PDF p. 77)

> "Suppose that you have a model for setting the side of the bet (long or short). You just need to learn the size of that bet, which includes the possibility of no bet at all (zero size). […] I call this problem meta-labeling because we want to build a secondary ML model that learns how to use a primary exogenous model. […] The ML algorithm will be trained to decide whether to take the bet or pass, a purely binary prediction. When the predicted label is 1, we can use the probability of this secondary prediction to derive the size of the bet, where the side (sign) of the position has been set by the primary model."

> "Binary classification problems present a trade-off between type-I errors (false positives) and type-II errors (false negatives). […] Meta-labeling is particularly helpful when you want to achieve higher F1-scores."

**Project mapping:** project filters like `ORB_G5`, `OVNRNG_*`, `VWAP_MID_ALIGNED` are **primary-model side-selectors** plus a **meta-labeling decision** (the filter decides binary trade/no-trade). The Ch 3.6 framing is the formal mapping for what we call "lane-overlay filters". Any new overlay filter pre-reg should be framed in meta-labeling terms (primary model = entry rule + RR target; secondary model = the filter's binary trade/no-trade prediction).

## § Ch 7 — Cross-Validation in Finance (verbatim load-bearing passages, PDF p. 130-138)

### § 7.3 Why k-fold CV fails in finance (verbatim, PDF p. 131)

> "By now you may have read quite a few papers in finance that present k-fold CV evidence that an ML algorithm performs well. Unfortunately, it is almost certain that those results are wrong. One reason k-fold CV fails in finance is because observations cannot be assumed to be drawn from an IID process. A second reason for CV's failure is that the testing set is used multiple times in the process of developing a model, leading to multiple testing and selection bias."

> "Leakage takes place when the training set contains information that also appears in the testing set. Consider a serially correlated feature X that is associated with labels Y that are formed on overlapping data: Because of the serial correlation, Xt ≈ Xt+1. Because labels are derived from overlapping datapoints, Yt ≈ Yt+1. By placing t and t + 1 in different sets, information is leaked. When a classifier is first trained on (Xt, Yt), and then it is asked to predict E[Yt+1|Xt+1] based on an observed Xt+1, this classifier is more likely to achieve Yt+1 = E[Yt+1|Xt+1] even if X is an irrelevant feature."

**Plain reading and project mapping:** this is the same class bug as our E2 break-bar look-ahead registry — a feature that "looks independent" but actually shares information across train/test partitions. The project's solution (Mode A holdout, `HOLDOUT_SACRED_FROM = 2026-01-01`, no parameter tuning against OOS) is a structurally simpler version of Ch 7's framing. LdP's purged k-fold goes further by purging *individual training observations* that overlap a test observation's window, which is more efficient than discarding entire time periods.

### § 7.4 Purged K-Fold CV — purging + embargo (verbatim, PDF p. 132-135)

> "One way to reduce leakage is to purge from the training set all observations whose labels overlapped in time with those labels included in the testing set. I call this process 'purging.' In addition, since financial features often incorporate series that exhibit serial correlation (like ARMA processes), we should eliminate from the training set observations that immediately follow an observation in the testing set. I call this process 'embargo.'"

> "We will determine that there is informational overlap between two observations i and j whenever Yi and Yj are concurrent (see Chapter 4, Section 4.3), in the sense that both labels are contingent on at least one common random draw. For example, [...] a label Yi = f[[ti,0, ti,1]] overlaps with Yj if any of the three sufficient conditions is met:
> 1. tj,0 ≤ ti,0 ≤ tj,1
> 2. tj,0 ≤ ti,1 ≤ tj,1
> 3. ti,0 ≤ tj,0 ≤ tj,1 ≤ ti,1"

> "For those cases where purging is not able to prevent all leakage, we can impose an embargo on training observations after every test set. […] A small value h ≈ .01T often suffices to prevent all leakage, as can be confirmed by testing that performance does not improve indefinitely by increasing k → T."

### § 7.4.3 The Purged K-Fold class — verbatim Snippet 7.3 (PDF p. 136)

> "Snippet 7.3 extends scikit-learn's KFold class to account for the possibility of leakages of testing information into the training set."

```python
class PurgedKFold(_BaseKFold):
    '''
    Extend KFold class to work with labels that span intervals
    The train is purged of observations overlapping test-label intervals
    Test set is assumed contiguous (shuffle=False), w/o training samples in between
    '''
    def __init__(self,n_splits=3,t1=None,pctEmbargo=0.):
        ...
```

**Project mapping:** if we build any ML or grid-search backtest against `orb_outcomes`, the train/test splitter MUST use this purging + embargo framework, NOT scikit-learn's bare `KFold`. The project's existing Mode A holdout is the *cruder* version of this (full chronological split with a 2026-01-01 cutoff). For ML feature-importance work or hyperparameter tuning, `PurgedKFold` is the right tool. The 1% embargo (`h ≈ 0.01T`) is the LdP-default; project should use this exact value unless the label-overlap window forces something longer.

### § 7.5 Bugs in sklearn's CV (verbatim, PDF p. 136-137)

> "You would think that something as critical as cross-validation would be perfectly implemented in one of the most popular ML libraries. Unfortunately that is not the case [...]
> 1. Scoring functions do not know classes_, as a consequence of sklearn's reliance on numpy arrays rather than pandas series
> 2. cross_val_score will give different results because it passes weights to the fit method, but not to the log_loss method"

**Project mapping:** if/when we use sklearn's CV utilities, use LdP's `cvScore` (Snippet 7.4) wrapper, not bare `cross_val_score`.

## § Ch 8 — Feature Importance (verbatim load-bearing passages, PDF p. 140-149)

### § 8.1-8.2 The motivation (verbatim, PDF p. 140-141)

> "One of the most pervasive mistakes in financial research is to take some data, run it through an ML algorithm, backtest the predictions, and repeat the sequence until a nice-looking backtest shows up. Academic journals are filled with such pseudo-discoveries, and even large hedge funds constantly fall into this trap. It does not matter if the backtest is a walk-forward out-of-sample. The fact that we are repeating a test over and over on the same data will likely lead to a false discovery. [...] It typically takes about 20 such iterations to discover a (false) investment strategy subject to the standard significance level (false positive rate) of 5%."

> **Snippet 8.1 — Marcos' First Law of Backtesting:** "Backtesting is not a research tool. Feature importance is."

**Project mapping:** this is the same lesson as Aronson 2007 § Ch 8-9 (6,402 rules / α=0.05 → 320 expected by chance / 320 found). LdP's quantification — *20 iterations → 1 false discovery at α=0.05* — is operationally tighter than Aronson's empirical demonstration. **20 backtests on the same data is the project's red-line** for any single research thread without pre-reg + multiplicity correction. Our `pre_registered_criteria.md` MinBTL bound (Bailey 2013) operationalizes this at the global level (no brute-force >300 trials).

### § 8.3.1 Mean Decrease Impurity (MDI) — verbatim (PDF p. 141-143)

> "Mean decrease impurity (MDI) is a fast, explanatory-importance (in-sample, IS) method specific to tree-based classifiers, like RF. [...] There are some important considerations you must keep in mind when working with MDI:
>
> 1. Masking effects take place when some features are systematically ignored by tree-based classifiers in favor of others. In order to avoid them, set max_features=int(1) when using sklearn's RF class. In this way, only one random feature is considered per level.
> 2. The procedure is obviously IS. Every feature will have some importance, even if they have no predictive power whatsoever.
> 3. MDI cannot be generalized to other non-tree based classifiers.
> 4. By construction, MDI has the nice property that feature importances add up to 1, and every feature importance is bounded between 0 and 1.
> 5. The method does not address substitution effects in the presence of correlated features. MDI dilutes the importance of substitute features [...] The importance of two identical features will be halved, as they are randomly chosen with equal probability.
> 6. Strobl et al. [2007] show experimentally that MDI is biased towards some predictor variables."

### § 8.3.2 Mean Decrease Accuracy (MDA) — verbatim (PDF p. 143)

> "Mean decrease accuracy (MDA) is a slow, predictive-importance (out-of-sample, OOS) method. First, it fits a classifier; second, it derives its performance OOS according to some performance score (accuracy, negative log-loss, etc.); third, it permutates each column of the features matrix (X), one column at a time, deriving the performance OOS after each column's permutation. The importance of a feature is a function of the loss in performance caused by its column's permutation. Some relevant considerations include:
> 1. This method can be applied to any classifier, not only tree-based classifiers.
> 2. MDA is not limited to accuracy as the sole performance score. […]
> 3. Like MDI, the procedure is also susceptible to substitution effects in the presence of correlated features.
> 4. Unlike MDI, it is possible that MDA concludes that all features are unimportant. That is because MDA is based on OOS performance.
> 5. The CV must be purged and embargoed, for the reasons explained in Chapter 7."

### § 8.4.1 Single Feature Importance (SFI) — verbatim (PDF p. 144-145)

> "Single feature importance (SFI) is a cross-section predictive-importance (out-of-sample) method. It computes the OOS performance score of each feature in isolation. A few considerations:
> 1. This method can be applied to any classifier, not only tree-based classifiers.
> 2. SFI is not limited to accuracy as the sole performance score.
> 3. Unlike MDI and MDA, no substitution effects take place, since only one feature is taken into consideration at a time.
> 4. Like MDA, it can conclude that all features are unimportant, because performance is evaluated via OOS CV.
> The main limitation of SFI is that a classifier with two features can perform better than the bagging of two single-feature classifiers."

**Plain reading and project mapping:** these three methods solve three different questions:

| Method | Question answered | Use case in project |
|--------|-------------------|---------------------|
| MDI | "Which features did the model use internally?" | Diagnostic for our DAGs / tree ensembles in research/ when we want to understand model behavior. IS only. |
| MDA | "Which features cause OOS performance to drop when removed?" | The most operationally relevant. For overlay-filter pre-reg, an MDA permutation test on the candidate filter against `orb_outcomes.pnl_r` gives the right framing. Use `PurgedKFold`. |
| SFI | "Which features have signal in isolation, irrespective of substitution?" | The right framing when ranking a CANDIDATE FEATURE LIST without commitment to a particular classifier. Often the first-pass triage. |

For project-side feature-importance work the recommended sequence is: SFI to triage → MDA on top-K survivors with `PurgedKFold` → economic interpretation.

### § 8.4.2 Orthogonal features + PCA validation (verbatim, PDF p. 145-147)

> "An orthogonalization procedure such as principal components analysis (PCA) does not prevent all substitution effects, but at least it should alleviate the impact of linear substitution effects."

> "PCA has ranked features without any possible overfitting in a classification sense. When your MDI, MDA, or SFI analysis selects as most important (using label information) the same features that PCA chose as principal (ignoring label information), this constitutes confirmatory evidence that the pattern identified by the ML algorithm is not entirely overfit. If the features were entirely random, the PCA ranking would have no correspondence with the feature importance ranking."

> "I find it useful to compute the weighted Kendall's tau between the feature importances and their associated eigenvalues (or equivalently, their inverse PCA rank). The closer this value is to 1, the stronger is the consistency between PCA ranking and feature importance ranking."

**Project mapping:** the PCA-MDI consistency check is a *cheap* anti-overfitting diagnostic — if our most-important `daily_features` columns under MDA / SFI are NOT the same as the largest-variance principal components, that's potential overfitting evidence. The weighted Kendall's tau gives a single scalar to report. Worth adding to any feature-importance audit as a sanity check.

### § 8.5 Parallelized vs. Stacked feature importance (verbatim, PDF p. 148)

> "First, for each security i in an investment universe i = 1, …, I, we form a dataset (Xi, yi), and derive the feature importance in parallel. […] Features that are important across a wide variety of instruments are more likely to be associated with an underlying phenomenon, particularly when these feature importances exhibit high rank correlation across the criteria."

> "A second alternative is what I call 'features stacking.' It consists in stacking all datasets {(X̃i, yi)}i=1,…,I into a single combined dataset (X, y) [...] Under this approach, the classifier must learn what features are more important across all instruments simultaneously, as if the entire investment universe were in fact a single instrument."

> "I usually prefer features stacking, not only for features importance but whenever a classifier can be fit on a set of instruments [...] That reduces the likelihood of overfitting an estimator to a particular instrument or small dataset."

**Project mapping:** mirrors the project's `pooled-finding-rule.md` per-cell-breakdown requirement. LdP recommends *stacking* by default (treat MGC + MNQ + MES as one universe with instrument as a feature). For us this maps to **discovery should default to cross-instrument feature stacking with per-instrument breakdown reported alongside**. That's already the project's posture; LdP provides the ML-side rationale.

## § Ch 12 — Combinatorial Purged Cross-Validation (verbatim load-bearing passages, PDF p. 188-191)

### § 12.2 The Walk-Forward Method and its three pitfalls (verbatim, PDF p. 188-189)

> "The most common backtest method in the literature is the walk-forward (WF) approach. WF is a historical simulation of how the strategy would have performed in past. […] WF suffers from three major disadvantages: First, a single scenario is tested (the historical path), which can be easily overfit (Bailey et al. [2014]). Second, WF is not necessarily representative of future performance, as results can be biased by the particular sequence of datapoints. […] The third disadvantage of WF is that the initial decisions are made on a smaller portion of the total sample."

**Plain reading and project mapping:** our Mode A holdout (`HOLDOUT_SACRED_FROM = 2026-01-01`) is a walk-forward split — IS up to 2026-01-01, OOS after. LdP names the three weaknesses explicitly. The project's response is the OOS power floor (`research/oos_power.py`) and the Pathway-A/B + dir_match requirement; these mitigate disadvantages 2 and 3 but not 1 (single path). CPCV addresses disadvantage 1 directly.

### § 12.4 The Combinatorial Purged Cross-Validation method — verbatim, PDF p. 190

> "In this section I will present a new method, which addresses the main drawback of the WF and CV methods, namely that those schemes test a single path. I call it the 'combinatorial purged cross-validation' (CPCV) method. Given a number φ of backtest paths targeted by the researcher, CPCV generates the precise number of combinations of training/testing sets needed to generate those paths, while purging training observations that contain leaked information."

### § 12.4.1 Combinatorial splits — the formula (verbatim, PDF p. 190)

> "Consider T observations partitioned into N groups without shuffling, where groups n = 1, …, N − 1 are of size ⌊T/N⌋, the Nth group is of size T − ⌊T/N⌋(N − 1), and ⌊.⌋ is the floor or integer function. For a testing set of size k groups, the number of possible training/testing splits is
>
> C(N, N−k) = ∏ᵢ₌₀^(k−1) (N−i) / k!
>
> Since each combination involves k tested groups, the total number of tested groups is k · C(N, N−k). And since we have computed all possible combinations, these tested groups are uniformly distributed across all N (each group belongs to the same number of training and testing sets). The implication is that from k-sized testing sets on N groups we can backtest a total number of paths
>
> φ[N, k] = k/N · C(N, N−k) = ∏ᵢ₌₁^(k−1) (N−i) / (k−1)!
>
> Figure 12.1 illustrates the composition of train/test splits for N = 6 and k = 2. There are C(6, 4) = 15 splits, indexed as S1, …, S15. […] Each group forms part of φ[6, 2] = 5 testing sets, therefore this train/test split scheme allows us to compute 5 backtest paths."

**Plain reading and project mapping:** CPCV runs ALL combinations of training/testing splits over N groups of size T/N each. Each observation is therefore in MANY test sets (versus 1 in WF, or 1 in standard k-fold). The output is a distribution of φ backtest paths, NOT a point estimate. This is the key innovation — it converts WF's single-path overfitting bound into a multi-path distribution.

For our project's typical setup (7+ years of training data, 4+ months of OOS), N=6 splits with k=2 yields φ=5 alternative backtest paths — a substantial improvement over the single Mode A path. **The right tool for the project's "OOS too short to refute" situations** that the bull-short-avoidance incident exposed.

## How this paper SHOULD and SHOULD NOT be used in our project

### LEGITIMATE uses

1. **Primary citation for CPCV** in any pre-reg, audit-result, or methodology rule that invokes CPCV as the right OOS gate. Cite as "AFML 2018 § 12.4". The 2020 *Machine Learning for Asset Managers* sampler is a secondary citation only.
2. **Primary citation for Purged K-Fold CV** (`PurgedKFold` class, embargo, purging by label-overlap interval) — Ch 7 § 7.4. Use when any project-side ML or grid-search work needs a CV splitter. The bare scikit-learn `KFold` is forbidden for financial time-series.
3. **Primary citation for the triple-barrier labeling method** (Ch 3 § 3.4). All project research that constructs custom labels should adopt this framing; the project's `orb_outcomes` schema is already structurally compatible.
4. **Primary citation for meta-labeling** (Ch 3 § 3.6) — for framing the filter-as-binary-meta-classifier conceptual model. Project overlay filters (ORB_G5, OVNRNG_*) are already implicit meta-labelers; this gives them a name.
5. **Primary citation for MDI / MDA / SFI feature importance** (Ch 8 §§ 8.3-8.4). All three are appropriate for different research questions; project-side feature audits should triage via SFI → confirm via MDA with `PurgedKFold`.
6. **Marcos' First Law** ("Backtesting is not a research tool. Feature importance is.") — cite when discouraging brute-force backtest iteration, as a one-line alternative to citing Aronson 2007's 6,402-rule case study.
7. **The 20-iterations-to-discovery operational warning** (Ch 8 § 8.1) — tighter operational analog to MinBTL. Pair with Bailey 2013 in pre-reg gates.

### ILLEGITIMATE uses

1. **As replacement for our pre-reg discipline.** CPCV gives more backtest paths but does not solve multiple-testing-bias if the researcher re-runs CPCV with different feature sets until something sticks. The project's pre-reg + MinBTL bound + BH-FDR survives CPCV adoption.
2. **As justification for any specific feature set.** AFML is methodology, not specific finance results. The "informative features" in LdP's synthetic experiments are exactly that — synthetic. Do not transfer his feature importance rankings to our problem.
3. **As substitute for understanding the underlying problem.** ML feature importance is a tool, not a substitute for domain knowledge of the ORB mechanism, microstructure, or session structure. Stack feature importance with mechanism priors (Tolušić 2026, Yordanov 2026, Howard 2026, Harris 2002) — do not run feature importance in isolation.
4. **As proof that ML methods work on the project's data.** LdP's worked examples are on synthetic data or simplified equity-prediction problems. Any project-side ML application must be validated on `gold.db` with `HOLDOUT_SACRED_FROM = 2026-01-01` discipline, not assumed to work because LdP showed it works on his data.

### Pre-registration notes if pursuing AFML-grounded work

Per `pre_registered_criteria.md` and `.claude/rules/backtesting-methodology.md`:
- **CPCV pre-reg:** any pre-reg using CPCV must declare N (number of groups) and k (test-set size in groups), and report all φ paths' realized metrics, not just the median or mean.
- **Feature-importance pre-reg:** any feature-importance study must declare which of MDI/MDA/SFI it uses, why, and report all three for top-K survivors as a cross-check.
- **Triple-barrier pre-reg:** any new label construction must declare the three barrier widths (profit-take multiplier, stop multiplier, time-out bars) and report the path-dependence stats (% of labels touched at each barrier).
- **Meta-labeling pre-reg:** any overlay-filter pre-reg should be framed in meta-labeling terms (primary model = entry rule; secondary model = filter binary). This is a re-framing, not a new methodology, but makes the false-positive vs false-negative trade-off explicit.
