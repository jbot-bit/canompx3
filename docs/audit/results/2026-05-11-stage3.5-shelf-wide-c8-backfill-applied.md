## Limitations

- The JSON output file `c8_backfill_applied.json` was polluted by
  logger output writing to stdout (instead of stderr), so the JSON file
  itself is not parseable. The DB write nonetheless completed (verified
  via direct query). The logger-stdout issue is a separate
  infrastructure cleanup, not a backfill correctness issue.
- The C8 evaluator's OOS window is `trading_day >= HOLDOUT_SACRED_FROM`
  (2026-01-01 onwards). Rows with very recent promotion may have OOS
  N too low to clear the N >= 30 CLT heuristic; those flip to
  INSUFFICIENT_N_PATHWAY_A_PASS_THROUGH (Pathway A permissive).
- 137 rows are pass-through (INSUFFICIENT_N + NO_OOS_DATA). These are
  not "edge fails"; they are evidence gaps. Stage 4 should treat these
  as a separate class from PASSED and from FAILED.
- 224 rows now carry explicit FAIL (NEGATIVE_OOS_EXPR + FAILED_RATIO).
  These have always been C8-blocked at deployability; the only change
  is the verdict is now visible as a status string instead of `c8_missing`.
