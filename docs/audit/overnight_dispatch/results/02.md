**VERDICT: PARTIAL — finding is REAL but significantly MITIGATED by a later code layer**

**EVIDENCE**

`contract_resolver.py:52` reads `acct = accounts[0]` — no matching against any configured account name, ID, or profile field. Takes the first account the broker API returns.

But the full live-launch path mitigates "silent wrong-account routing":
- `run_live_session.py:1063-1066` — `--account-id` CLI arg, `default=None`.
- Multi-copy (`n_copies > 1`) routes through `resolve_all_account_ids()` + `_select_primary_and_shadow_accounts()` (`run_live_session.py:945-991`), which validates `requested_account_id` against the discovered list and raises if not found.
- Single-copy (the C11 `topstep_50k_mnq_auto` deployment): `run_live_session.py:1399-1403` passes `account_id=args.account_id`; `session_orchestrator.py:646-647` only calls `resolve_account_id()` if `account_id is None or == 0`. With `--account-id` unset, `args.account_id` is `None` → `resolve_account_id()` fires → `accounts[0]`.

**SEVERITY: REAL but conditional.** One active account on the API key → `accounts[0]` is correct. Two+ linked accounts (second eval / copy account) → `accounts[0]` may not be the C11-profiled account. API response ordering is undocumented, not guaranteed stable.

**FIX SKETCH:** `AccountProfile` should carry optional `account_id: int | None`. `run_live_session.py` auto-populates `args.account_id` from the profile before constructing `SessionOrchestrator`, eliminating the blind `accounts[0]` fallback for all profile-bound launches. Extend `_select_primary_and_shadow_accounts` explicit matching to the single-copy path.

**SKIPPED:** Tradovate / Rithmic `resolve_account_id` not traced — same blind-index-0 pattern may exist there (residual risk).

**Decision: VERIFY_MORE** — safe only if operator's API key has exactly ONE active account. Confirm via `resolve_all_account_ids()` count before arming live.
