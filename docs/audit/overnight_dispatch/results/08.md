## MED #1 partial-fill bracket — verification result (2026-06-06)

**Status: OPEN — undocumented by ProjectX; speculative resize code NOT written (would be a band-aid).**

Primary-source grounding done; none resolves the qty-follows-fill question:
- Official API ref ([order-place](https://gateway.docs.projectx.com/docs/api-reference/order/order-place/))
  documents only `stopLossBracket`/`takeProfitBracket` as `{ticks, type}` — **no size field on
  the bracket objects**, and is silent on partial-fill behavior.
- Help-center bracket/position pages now 301-redirect to Topstep marketing (no content).
- Community SDK ([project-x-py](https://project-x-py.readthedocs.io/en/stable/api/trading.html))
  is also silent on partial-fill bracket sizing.

Repo-code facts (grounded):
- `projectx/order_router.py:323-329` — AutoBracket creates **separate child orders** `entry_id+1`
  (SL) / `entry_id+2` (TP), not a single position-attached OCO. So whether they track the *position*
  (filled qty) or the *order* (requested qty) is a server-side ProjectX behavior, unobservable from
  our spec which carries **no qty on the bracket** (`build_bracket_spec` returns only tick offsets).
- `session_orchestrator.py` fill handling validates fill **price** (`_validate_fill_price:1320`) but
  **never reads back filled quantity** vs ordered quantity — confirmed no resize path exists.

**Why dormant today:** the live edge book is clamped to 1 micro/lane (`max_contracts=1`). A 1-lot
market order is all-or-nothing — it cannot partially fill. The risk materializes **only when
`execution_qty>1`** (a contract-multiple execution map, or the clamp-lift income plan). So this is a
**clamp-lift prerequisite**, same gate-class as rows 03/07 — not an arm blocker at current sizing.

**Verification path (do before any execution_qty>1):**
1. Place a multi-lot bracket entry on a **demo/sim** account, force a partial fill (thin book or
   small marketable-limit), then `searchOpen` the `entry_id+1`/`entry_id+2` children and read their
   `size`. If child size == filled qty → ProjectX tracks the position; no code needed.
   If child size == ordered qty → a residual unprotected lot exists → add post-fill resize.
2. Alternatively, query ProjectX/Topstep support for AutoBracket partial-fill semantics in writing.

**No code written**: writing a resize-on-partial path on a *guess* risks double-protection or a
cancel/resubmit race that is worse than the dormant gap. Gated behind the empirical probe above.
